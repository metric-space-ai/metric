# Hero Application Visualization Plan

This plan turns METRIC hero applications into reproducible public evidence:
real datasets, native C++ evidence generation, and visual pages that make the
finite metric-space advantage visible without relying on claim-only tables.

## Non-Negotiables

- Real algorithmic work stays in C++.
- Python, JavaScript, or shell may only format exported evidence or render
  static assets. They must not compute metric results.
- No new C++ runtime dependencies.
- No prompt history, agent notes, or implementation intentions may appear in
  user-facing docs.
- Every public number on the project site must be generated by a checked-in C++
  executable or a checked-in static asset with documented source.
- Vector baselines must be named as baselines, not as strawman insults.
- MGC is a dependence test between spaces, not a metric.
- PHATE and autoencoder workflows are derived-space workflows over an existing
  finite metric space, not a standalone ML toolbox.

## First Hero Package: Process-Curve Condition Monitoring

### User Story

An engineer has process windows from machines or industrial systems. The windows
are not naturally fixed feature vectors because timing shifts and local
stretching matter. METRIC should show that a finite metric space over process
curves can recover the right operating condition where a padded point-vector
comparison fails.

### Existing Inputs

- Source dataset: UCR Time Series Anomaly Detection datasets 2021.
- Source license: CC BY 4.0.
- Source DOI: `10.6084/m9.figshare.26410744.v1`.
- Local C++ fixture:
  `examples/engine/process_curve_external_gallery.cpp`.
- Local data slices:
  `examples/engine/assets/process_curve_power_demand_gallery.csv`,
  `examples/engine/assets/process_curve_internal_bleeding_gallery.csv`.
- Existing visual assets:
  `docs/examples/assets/process-curve-power-demand-gallery.svg`,
  `docs/examples/assets/process-curve-power-demand-report.svg`,
  `docs/examples/assets/process-curve-power-demand-query-winners.svg`,
  `docs/examples/assets/process-curve-internal-bleeding-gallery.svg`,
  `docs/examples/assets/process-curve-internal-bleeding-report.svg`.
- Existing evidence doc:
  `docs/examples/process-curve-external-gallery.md`.

### Current Evidence Contract

The existing executable already asserts:

- 2 domains.
- 48 source records.
- 16 downsampled queries.
- 8/8 metric-space matches on Power Demand.
- 8/8 metric-space matches on Internal Bleeding.
- 8/8 padded-vector mismatches on Power Demand.
- 8/8 padded-vector mismatches on Internal Bleeding.
- 1152 dense source-space evaluations.

This is strong enough for a first public hero page, but the evidence is still
presented too much like a text report. The next step is to make the evidence
visual and inspectable.

## Required C++ Evidence Export

Extend `examples/engine/process_curve_external_gallery.cpp` with an optional
export mode:

```bash
build/core/examples/engine/engine_process_curve_external_gallery \
  --export-dir docs/examples/assets/process-curve-external
```

Default behavior must remain unchanged: running the executable without
arguments prints the current deterministic text output and returns nonzero on
failed assertions.

The export mode must create this directory and write these files:

### `summary.csv`

One row per domain.

Required columns:

- `domain`
- `source`
- `license`
- `records`
- `queries`
- `metric_correct`
- `metric_total`
- `vector_mismatches`
- `vector_total`
- `average_metric_margin`
- `dense_evaluations`

### `records.csv`

One row per source process window.

Required columns:

- `domain`
- `record_index`
- `record_id`
- `source_label`
- `sample_count`
- `values`

`values` must be a space-separated list of finite numbers so the asset remains
simple to parse by static renderers.

### `queries.csv`

One row per query.

Required columns:

- `domain`
- `query_index`
- `query_id`
- `expected_label`
- `metric_winner_id`
- `metric_winner_label`
- `metric_distance`
- `vector_winner_id`
- `vector_winner_label`
- `vector_distance`
- `metric_margin`
- `metric_correct`
- `vector_mismatch`
- `query_values`

### `distances.csv`

One row per domain, source record pair, and representation.

Required columns:

- `domain`
- `representation`
- `row_index`
- `row_id`
- `column_index`
- `column_id`
- `distance`

`representation` must be either `metric_space` or `padded_vector_baseline`.

### `query-winners.csv`

One row per query and compared winner curve.

Required columns:

- `domain`
- `query_id`
- `curve_role`
- `curve_id`
- `curve_label`
- `distance`
- `values`

`curve_role` must be one of `query`, `metric_winner`,
`vector_baseline_winner`.

## Visual Copy Rules

Use these words consistently:

- record
- finite metric space
- metric value
- query
- source space
- derived space
- baseline
- dependence test
- representative

Avoid these as primary public labels:

- distance provider
- matrix cache
- distance matrix snapshot
- AI pipeline
- vector alternative
- generic embedding workflow

When a distance is explained, phrase it as a metric value between two records.
If the cost interpretation is useful, say it is an interpretation, not the
primitive definition.

## Commit And Push Rule

Commit only after the acceptance criteria pass. Use one focused commit:

```bash
git add \
  examples/engine/process_curve_external_gallery.cpp \
  docs/examples/process-curve-external-gallery.md \
  docs/examples/assets/process-curve-external \
  README.md \
  docs/site/index.html
git commit -m "Add visual process-curve hero evidence"
git push origin main
```

If unrelated local changes exist, do not include them.
