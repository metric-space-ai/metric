Finite metric-space computing

METRIC

METRIC makes arbitrary finite data computable by giving records a numeric transformation cost. A string, image, time series, graph, histogram, vector, or mixed industrial record becomes part of a finite metric space once the least-cost recoding path to every other record is defined.

Core idea Hero applications
Records Any finite typed data: curves, images, text, graphs, distributions, vectors, or mixed records.
+
d(a, b) One function for two records: the least-cost path for recoding a into b.

Implicit space The finite metric space exists before any matrix, graph, index, or embedding is materialized.

The user models meaning locally. METRIC handles the global computation: lazy distance calls, cache, matrix, graph, tree, sampling, correlation, entropy, and mappings.

One pairwise function defines the whole finite space.

The user does not build an all-with-all model. The user defines what it costs to turn one record into another. For a finite record set, that single binary function implicitly defines every pairwise relation. METRIC can then compute only the distances an operator needs, or materialize more structure when that is the better execution plan.

1 Keep native records A record remains a curve, string, image, graph, histogram, vector, or structured object. It is not flattened just to satisfy an algorithm.
2 Define d(a, b) The metric encodes the allowed transformation path and its costs: edit, alignment, transport, graph movement, vector displacement, or a composed domain rule.
3 Get (X, d) The finite set plus the metric is a finite metric space. It is a numerical object even when the records themselves are not numbers.
4 Let the framework execute METRIC routes each operator through lazy metric calls, cached distances, matrices, graphs, trees, sampling, or mappings.

Explicit structures are generated when computation needs them.

The finite metric space is not the same thing as a precomputed distance matrix. The matrix is only one representation. A graph, tree, cache, or learned map can represent the same underlying metric-space contract. This is why the user can focus on the pairwise metric while the engine takes responsibility for explicit computation.

Lazy metric calls

Operators can ask for d(a, b) only when they need a value. Large spaces do not have to start as dense matrices.

Matrix cache

Algorithms that need dense pairwise access can materialize distances and reuse them with explicit runtime diagnostics.

Graph and tree views

Neighborhood graphs and search trees expose local geometry for search, clustering, sparsification, and scalable traversal.

Mappings

When coordinates are useful, a mapping creates a derived metric space. The mapping is downstream of the source metric, not a substitute for it.

Vector spaces are included as the coordinate-cost case.

If the records are vectors and the metric is Euclidean distance, the recoding cost is just coordinate displacement. METRIC keeps that case. The generalization is that the cost function can also be edit cost, time-alignment cost, transport cost, graph cost, image-structure cost, or a weighted composition across fields.

The operators answer different questions.

Once the finite metric space exists, different operators have distinct meanings. The page keeps them separate because a correlation is not a distance and a mapping is not the source metric.

Record distance

Compares two records inside one domain: edit distance for text, TWED-style alignment for curves, EMD for distributions, or a composed metric for structured records.

Space distance

Compares whole finite geometries. RiemannianDistance uses induced distance or Laplacian matrices to measure distance between spaces.

Cross-space dependency

MGC uses two aligned distance matrices and returns a dependency statistic. It is correlation over paired spaces, not a distance function.

Entropy and monitoring

Entropy estimates local freedom in the metric geometry. That makes it useful for drift, novelty, and condition-monitoring diagnostics.

Representatives and outliers

Medoids, representative compression, DBSCAN-style noise, and neighbor graphs expose structure without first requiring coordinates.

Mappings

PCFA, PHATE-AE-style pipelines, and other mappings create derived spaces when coordinates, visualization, reconstruction, or out-of-sample transforms are needed.

C++ is the implementation surface.

Metrics, spaces, representations, operators, mappings, diagnostics, and PHATE-AE pipeline components are native C++. Python adapts user data to selected native surfaces.
