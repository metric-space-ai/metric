Generalized vector-space computation

METRIC

Vector algorithms start with coordinates. METRIC starts with records and a distance function. A vector space is one case of that model; a finite metric space covers strings, curves, histograms, mixed records, graphs, images, and vectors with one C++ engine surface.

The generalization

Question Vector space Finite metric space
What is a record? A coordinate row such as `[1.2, 0.4, 9.1]`. Any item with a defined distance: string, curve, histogram, image-like distribution, graph, structured record, or vector.
What gives geometry? A norm or vector distance such as Euclidean, cosine-derived, or Manhattan distance. An explicit domain metric `d(x, y)` that compares two records in their native form.
What changes? Every object must become coordinates before computation. Computation can happen before, after, or without vectorization.

One core idea: records plus metric equals space.

METRIC treats a dataset as a finite metric space when it has a finite set of records and a distance function. The distance function is the modeling decision. It defines which records are near, which groups are meaningful, which graph edges exist, and which mapping diagnostics are valid.

Records X = {x0, x1, ..., xn}

The dataset is finite. Records keep their domain type.

Metric d(x, y)

The distance compares two records and exposes the geometry used by operators.

Space (X, d)

The pair becomes the object that search, grouping, graph, diagnostic, and mapping code operates on.

The engine workflow is linear.

METRIC is not a catalog of unrelated algorithms. The public model is a sequence of choices over the same finite metric space.

1Choose recordsKeep the data in the domain where comparison is meaningful: strings, process curves, histograms, mixed records, or vectors.
2Choose the metricUse edit distance, transport cost, time alignment, a graph distance, Euclidean distance, or a composed domain metric.
3Create a metric spaceThe C++ engine stores stable record IDs, metric traits, metadata, and version state.
4Select representationRun implicitly, materialize a matrix cache, build a graph, use a tree index, or fit a mapping.
5Run operatorsFind neighbors, groups, representatives, outliers, graph structure, entropy, cross-space dependency, or mapped spaces.
6Inspect resultsResults keep source IDs, diagnostics, representation metadata, metric assumptions, warnings, and lineage.

Advantages and costs.

The benefit is not magic speed. The benefit is preserving the right geometry. The cost is that metric-space computation makes the distance function explicit and often more expensive than a fixed vector dot product.

Advantages
  • Native distances can preserve order, transport, alignment, structure, and mixed-field semantics.
  • Vector spaces remain available as a normal special case.
  • Representations are explicit: implicit lookup, dense matrix, graph, tree index, or mapped space.
  • Results can report assumptions, diagnostics, and source-record lineage.
Costs
  • A useful metric must exist; a weak metric produces weak results.
  • Pairwise distances can be expensive for large record sets.
  • Some indexes and algorithms require metric laws such as symmetry and triangle inequality.
  • METRIC is a native C++ library, not a hosted vector database or tensor training framework.

C++ is the implementation surface.

Metrics, spaces, representations, operators, mappings, diagnostics, and PHATE-AE pipeline components are native C++. Python adapts user data to selected native surfaces. 

#include <metric/distance.hpp>
#include <metric/engine.hpp>

#include <string>
#include <vector>

std::vector<std::string> records = {"metric", "metrics", "matrix", "tree"};
auto space = metric::make_space(records, metric::Edit<char>{});

auto nearest = metric::find_neighbors(
    space,
    std::string("metricks"),
    metric::count{2},
    metric::strategies::cover_tree{});
