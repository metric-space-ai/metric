Arbitrary records made numerically computable

METRIC

METRIC is a native C++ framework for computing with finite metric spaces. A record can be a vector, string, image, time series, graph, distribution, or mixed industrial object. The metric is the numerical bridge: it models the minimum cost path for recoding one record into another.

When it matters Use vectors directly when coordinates are the natural record. Use finite metric spaces when the data has structure that should stay visible: order, transport, shape, graph constraints, symbols, or mixed fields.

A finite metric space is how arbitrary records become calculable.

METRIC starts with a finite record set `X` and a metric `d`. The metric is not just a vague similarity score. It is the domain model of what it costs to change one record into another under the constraints that matter for the data. Once every pair of records has a numeric recoding cost, the set becomes a finite metric space `(X, d)`. Vectors are included as the simple case where the recoding cost is a coordinate displacement.

Records X = {x0, x1, ..., xn}

The dataset is finite. Records keep their native type and structure.

Recoding cost d(x, y) = min cost(path)

The metric is the least-cost transformation path from one record to another.

Numerical object (X, d)

The pair is the common substrate for search, grouping, entropy, graphs, correlation, mapping, and compression.

The distance function is the model of permitted transformations.

Coordinates are one way to obtain distances. METRIC also works when distance comes from edit operations, transport flow, time alignment, graph structure, image structure, or a composed industrial-domain metric. The engine evaluates those least-action costs and derives the numeric objects that algorithms actually consume.

Edit and alignment paths

Strings, event streams, and time series become computable when edits, gaps, warps, and time penalties define the allowed path from one record to another.

Transport and graph costs

Histograms, distributions, images, and structured objects can use transport or graph-ground costs instead of pretending that raw coordinates carry the right meaning.

Explicit representations

The finite space can be represented as a distance matrix, sparse graph, search tree, or dynamic metric lookup. These are execution structures, not replacements for the metric model.

Statistics and mappings

Entropy, intrinsic dimension, metric-space distances, cross-space correlation, embeddings, learned maps, and representatives can operate on distance geometry.

Metric spaces can be related in different ways.

A single analysis does not have to collapse every datatype into one record format. METRIC can build one finite metric space for one modality and another finite metric space for another modality, each with its own record container and metric. Then the question must be stated precisely. A metric-space distance asks how far two finite geometries are from each other. MGC asks whether two aligned spaces contain dependent structure. MGC is a correlation statistic, not a distance function. 

#include <metric/distance/d-spaced/Riemannian.hpp>
#include <metric/engine.hpp>

auto process_space = metric::make_space(process_records, metric::Euclidean<double>{});
auto quality_space = metric::make_space(quality_records, metric::Manhattan<float>{});

auto dependency = metric::correlate(process_space,
                                    quality_space,
                                    metric::strategies::mgc{}); // paired-space correlation

metric::RiemannianDistance<void, metric::Euclidean<double>> space_distance;
auto geometry_distance = space_distance(batch_a_records, batch_b_records); // space distance
Metric-space distance

`RiemannianDistance` represents each finite space through pairwise distances and measures the distance between those induced geometries.

Paired-space dependency

MGC uses two distance matrices over aligned observations and returns a dependency score. It answers whether structure in one space tracks structure in another.

Different record types

The left space can contain curves, vectors, strings, graphs, or structured records. The right space can use a different type and a different metric.

No joint embedding required

Both sides keep their native metric. Shared observation order is only needed for dependency analysis, not for every space-to-space distance question.

Use arbitrary datatypes as numerical objects.

Numbers are useful because software can compare, sort, group, approximate, map, and correlate them. METRIC gives arbitrary records the same kind of computational access through explicit recoding costs: strings can use edit distance, images can use transport or structural distance, time series can use alignment distance, graphs can use graph distance, and mixed records can combine those with ordinary numeric fields.

APer-type metricDefine one distance rule for each datatype or field: text, image, curve, histogram, graph, categorical state, scalar sensor value.
BMetric compositionCombine those distances into one record metric when a record contains multiple datatypes. Cross-type comparison is explicit: use a pair metric, recoding cost, or mapping when a string, image, or time series must be compared across type boundaries.
CShared analysisRun the same metric-space operators over the resulting space: nearest records, clusters, representatives, outliers, graph structure, correlations, embeddings, compression, and learned mappings.
d(record_a, record_b) =
  w_text  * edit(record_a.text, record_b.text)
+ w_image * transport(record_a.image, record_b.image)
+ w_curve * alignment(record_a.series, record_b.series)
+ w_num   * euclidean(record_a.sensors, record_b.sensors)

This is the generalization that makes multimodal embedding optional instead of mandatory. If an embedding is the best representation, it fits as a mapping. If native metrics are more faithful, analysis can begin without training one global embedding model first.

The engine workflow is linear.

The workflow is the same whether the records are vectors, strings, images, curves, or mixed domain objects. The metric-space abstraction is the common numerical layer.

1Choose typed recordsKeep the data in the domain where comparison is meaningful: strings, images, curves, histograms, graphs, categorical states, numeric fields, or vectors.
2Attach metricsUse edit distance for text, transport or structural distance for images, alignment distance for time series, Euclidean distance for vectors, and composed metrics for mixed records.
3Create a metric spaceThe C++ engine stores stable record IDs, metric traits, metadata, and version state for the resulting finite space.
4Select representationRun implicitly, materialize a matrix cache, build a graph, use a tree index, or fit a mapping.
5Run analysisFind neighbors, groups, representatives, outliers, graph structure, entropy, metric-space distance, cross-space dependency, or mapped spaces.
6Inspect resultsResults keep source IDs, diagnostics, representation metadata, metric assumptions, warnings, and lineage.

Advantages and costs.

The benefit is a broader numerical substrate, not automatic speed. The cost is that the metric has to be designed, evaluated, and sometimes materialized.

Advantages
  • Arbitrary records become numerically computable once a meaningful recoding-cost metric is defined.
  • Vector datasets remain first-class; they are finite metric spaces with coordinate-derived distances.
  • Strings, images, curves, histograms, structured records, and vectors can participate in one analysis model.
  • Heterogeneous records can be analyzed without first training one global multimodal embedding network.
  • Native distances can preserve order, transport, alignment, structure, and mixed-field semantics.
  • Vector spaces remain available as a normal special case.
  • Representations are explicit: implicit lookup, dense matrix, graph, tree index, or mapped space.
Costs
  • A useful least-cost transformation model must exist for each type, field, or cross-type comparison that the analysis uses.
  • Composed metrics need explicit weights and assumptions.
  • When a natural vector representation already solves the problem, direct vector methods can be simpler.
  • Pairwise distances can be expensive for large record sets.
  • Some indexes and algorithms require metric laws such as symmetry and triangle inequality.
  • METRIC is a native C++ library, not a hosted vector database or tensor training framework.

Hero applications and evidence.

The checked-in C++ reports are small deterministic fixtures. They demonstrate the analysis pattern: keep the domain metric, derive numerical structures from distances, and only map to coordinates when that mapping is the right representation. The production direction is condition monitoring, time-series anomaly detection, cross-space dependency discovery, mixed-record analysis, and native metric-space mapping.

Condition monitoring entropy

Entropy is a diagnostic over a finite metric space. It uses local distance geometry to quantify structural freedom, useful for monitoring drift, regime changes, and novelty in process windows.

C++ is the implementation surface.

Metrics, spaces, representations, operators, mappings, diagnostics, and PHATE-AE pipeline components are native C++. Python adapts user data to selected native surfaces. 

#include <metric/distance.hpp>
#include <metric/engine.hpp>

#include <string>
#include <vector>

std::vector<std::string> records = {"metric", "metrics", "matrix", "tree"};
auto space = metric::make_space(records, metric::Edit<char>{});

auto nearest = metric::find_neighbors(
    space,
    std::string("metricks"),
    metric::count{2},
    metric::strategies::cover_tree{});
