Finite metric-space computing

METRIC

METRIC makes arbitrary finite data computable by giving records a numeric metric. A string, image, time series, graph, histogram, vector, or mixed industrial record becomes part of a finite metric space once a suitable built-in, configured, or custom record metric is selected. The framework then hides the mechanics of how that space is evaluated, sampled, mapped, embedded, and analyzed.

Core idea Application workflows

One pairwise function defines the whole finite space.

The user does not build an all-with-all model, choose a vector dimension, or design a global embedding first. The user chooses a metric from METRIC's library, configures it, or supplies a domain-specific metric when needed. For a finite record set, that single binary function implicitly defines every pairwise relation. METRIC decides how much of that geometry must be evaluated for each operator.

Mathematical principle

Finite records plus a true metric form a finite metric space. The usual metric laws still apply.

Engine components

MetricSpace, record helpers, metric catalog entries, storage/index views, stats, modify components, native solvers, results, artifacts, and diagnostics make the principle executable.

Implementations

Concrete metrics, estimators, strategies, presets, demos, and paper references sit behind those components.

1 Keep native records A record remains a curve, string, image, graph, histogram, vector, or structured object. It is not flattened just to satisfy an algorithm.
2 Choose d(a, b) The metric can be built in, configured, or custom. It assigns the smallest admissible relation value for a pair: edit, alignment, transport, graph movement, vector displacement, or a composed domain rule.
3 Get (X, d) The finite set plus the metric is a finite metric space. It is a numerical object even when the records themselves are not numbers.
4 Let METRIC execute The framework routes each question through lazy metric calls, distance tables, graphs, trees, local approximations, sampling, or mappings.

METRIC decides how to compute the space.

The finite metric space is not the same thing as a precomputed distance matrix. The matrix is only one computed view. A graph, tree, table, local Euclidean approximation, sampled metric access, or fitted map can serve the same source metric. The user focuses on the metric; the engine takes responsibility for explicit computation and reports what it chose.

Lazy metric calls

Operators can ask for d(a, b) only when they need a value. Large spaces do not have to start as dense matrices.

Distance table

Algorithms that need dense pairwise access can materialize metric values and reuse them with explicit diagnostics.

Graph and tree views

Neighborhood graphs, search trees, and local coordinate views expose local geometry for search, clustering, sparsification, and scalable traversal.

Mappings

When coordinates are useful, a mapping creates a result space. The mapping is downstream of the source metric, not a substitute for it.

Time series and curves

TWED, ERP, Hausdorff, and admitted curve or set metrics encode alignment, deletion, gap, and geometric transformation costs without flattening the signal first. Frechet remains experimental until its exact domain is admitted.

Distributions and images

Wasserstein, total variation, Hellinger, Kolmogorov-Smirnov, and square-root Jensen-Shannon metric variants model movement or distributional recoding costs.

Read non-metric warnings

Vector spaces are included as the coordinate-cost case.

If the records are vectors and the metric is Euclidean distance, the recoding cost has a closed coordinate formula. METRIC keeps that case. The generalization is that the cost function can also be computed by algorithms: edit dynamic programs, time-alignment costs, transport optimization, graph shortest paths, image-structure costs, or weighted compositions across fields. Local Euclidean approximations remain useful, but METRIC does not force the user to choose the approximation dimension before the metric-space computation can start.

One mixed record

A machine-state record can combine sensor vectors, status symbols, maintenance text, spectral histograms, and process curves through a composed metric.

Multiple spaces

A process curve space and a quality measurement space can remain separate while their geometries are compared or correlated.

No forced joint embedding

A neural multimodal embedding or high-dimensional vector surrogate is optional. It is a mapping when useful, not a prerequisite for analysis.

Embedding-aware analysis

Embeddings become result spaces with lineage, preservation diagnostics, reconstruction support, and out-of-sample behavior instead of opaque preprocessing.

Native C++ execution

The core algorithms and metric math belong in C++. Python and other languages are adapters to the native surface.

Framework components answer different questions.

Once the finite metric space exists, different components answer distinct questions. Entropy, correlation, sampling, and transformation are separate from the record metric: a correlation is not a distance and a mapping is not the source metric.

Record distance

Compares two records inside one domain: edit distance for text, TWED-style alignment for curves, EMD for distributions, or a composed metric for structured records.

Space distance

Compares whole finite geometries. RiemannianDistance uses induced distance or Laplacian matrices to measure distance between spaces.

Correlation between spaces

MGC uses two aligned distance matrices and returns a correlation statistic. It is not a distance function.

Entropy and monitoring

Entropy estimates local freedom in the metric geometry. That makes it useful for drift, novelty, and condition-monitoring diagnostics.

Sampling

Sampling and metric-space walks select records or traversals that summarize finite geometry without changing the authoritative metric.

Representatives and outliers

Medoids, representative compression, DBSCAN-style noise, and neighbor graphs expose structure without first requiring coordinates.

Mappings

PCFA, PHATE-AE-style presets, and other mappings create result spaces when coordinates, visualization, reconstruction, or out-of-sample transforms are needed.
