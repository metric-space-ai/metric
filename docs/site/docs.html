Technical Documentation

Finite metric spaces as native C++ numerical objects.

METRIC works with records and explicit metrics. The core engine owns the metric-space model; bindings adapt external language data to that native implementation.

Finite Metric Spaces

A finite metric space is a finite record set X = {x0, x1, ..., xn} plus a metric d(x, y). The metric gives the dataset its geometry. Engine components document when they require non-negativity, identity, symmetry, and the triangle inequality.

Records do not need to be vectors. METRIC can use strings with edit distance, histograms with transport distance, process curves with alignment distance, mixed records with composed domain penalties, or vectors with Euclidean distance.

Engine Model

Records + Metric -> MetricSpace -> Stats / Modify / Storage -> Result
MetricSpaceStores records, stable IDs, metric traits, metadata, and version state.
Storage and indexesRun the same space implicitly, through a distance table, graph, tree index, or mapped space.
Stats and modifyCompute neighbors, groups, representatives, graph diagnostics, entropy, MGC, mappings, and reductions.
ResultPreserves IDs, strategy metadata, representation metadata, warnings, diagnostics, and lineage.

Metric Discovery

Metric Discovery guides users from their data type to an implemented metric, an admission candidate, or a quarantined non-metric warning before they write a custom function. Each documented metric states its record domain, meaning, metric-law note, example, and public status: stable, experimental, or quarantine.

Discovery GuideChoose metrics by domain: vectors, strings, sets, curves, distributions, graphs, images, and structured records.
True Metric CatalogOnly exact computations and valid domains with proven metric laws enter guarantee-dependent engine paths.
Admission RulesEvery candidate records triangle-inequality evidence, property tests, numerical edge cases, and C++ implementation ownership.
Non-Metric WarningsRaw cosine dissimilarity, KL divergence, raw Jensen-Shannon divergence, default DTW, raw SSIM, generic Tanimoto names, and squared CvM statistics are documented separately from metric variants.

The source documents are maintained under docs/metrics/.

C++ API

#include <metric/metric/catalog.hpp>
#include <metric/engine.hpp>

#include <string>
#include <vector>

std::vector<std::string> records = {"metric", "metrics", "matrix", "tree"};
auto space = mtrc::make_space(records, mtrc::Edit<char>{});

auto neighbors = mtrc::stats::search::find_neighbors(
    space,
    std::string("metricks"),
    mtrc::count{2});

The detailed C++ reference is maintained in docs/api/cpp.md.

Bindings

Python exposes selected native surfaces for user records, result objects, and adapters. Algorithmic work belongs to the C++ implementation. Binding documentation is maintained in docs/api/python.md. 

from metric import Edit, Space

records = ["cat", "cot", "coat", "dog"]
space = Space(records, Edit())
print(space.neighbors("cut", k=2))

Application Workflows

The checked examples are C++ workflows that start from native record types and an admitted or explicitly documented metric computation. They demonstrate modeling patterns for monitoring, distribution recoding, mixed records, dependency analysis, sampling, and metric-aware transformations.

Condition MonitoringProcess windows remain time-series records. Alignment metrics support entropy, outlier, representative, and dependency analysis over the process geometry.
Distribution RecodingHistograms and image-like records use movement cost, so shifted structure can remain near even when coordinate overlap is weak.
Mixed Industrial RecordsSensor vectors, states, text, histograms, and curves can be compared with one composed metric instead of a forced multimodal embedding.
Metric-Aware EmbeddingsPHATE-AE-style pipelines are native engine compositions that produce derived coordinate spaces with provenance and diagnostics.

The documented outputs are in docs/examples/engine-demo-outputs.md.
